## 1. 领域哲学与心智模型

**一句话**：给独立律师 / 中小律所一个不臃肿、能真正用起来的案件全生命周期管理系统。

**核心心智模型**：律师脑子里只有三件事——**做案子 / 看客户 / 看钱**。系统的一切围绕这三条主线，而不是按数据表堆菜单。

**主线流程**：
```
收案登记 → 利益冲突检索 → 审批转正式案件 → 多程序办理
   → 过程管理（期限/开庭/沟通/材料/卷宗/财务）→ 结案归档 → 数据导出
```

**设计原则**（重写时应继承）：
- 关键操作 ≤ 2 次点击（新建收案、找案件、看下次开庭）。
- 从问题倒推交互，不照搬传统案管软件的菜单。例如"开庭"不是独立大菜单，而是案件详情的一部分。
- 数据可导出（CSV/PDF/XLSX）——律所要能随时把数据拿走，不被锁定。
- 自部署优先，数据在律所自己手里。

## 2. 角色与权限模型

### 2.1 五种角色

| 角色 | 含义 | 可见范围 |
|---|---|---|
| `ADMIN` | 系统管理员 | 全部；含用户管理、审计、导出 |
| `PRINCIPAL_LAWYER` | 主任律师 / 合伙人 | 全部案件（管理/审批/统计视角） |
| `LAWYER` | 经办律师 | 自己主办 + 作为成员参与的案件 |
| `ASSISTANT` | 律师助理 | 按案件成员授权的案件 |
| `FINANCE` | 财务 | 所有案件的财务字段可见可编辑，案件正文只读 |

### 2.2 权限不变量（重写必须遵守）

1. **访问控制在服务端强制**，不能只靠 UI 隐藏——所有读写都要在后端按角色过滤/校验。
2. **管理角色的"全所可见"仅用于管理/审批/统计视角**。但创建业务单据（用印、快递、保全、开票）、以及手动关联短信/案件时，**必须限定为当前用户主办或参与的案件**——不能因为是管理员就能把任意案件挂上。
3. **"是否本人创建"不作为权限判断依据**——权限基于角色 + 案件成员关系（主办/协办/助理），不基于 creator。
4. **身份证号/统一社会信用代码按角色脱敏**：仅 ADMIN / 案件主办 / 协办可见明文，其他角色看掩码。⚠️ *原实现未做，见第 9 节。*

## 3. 核心领域模型：三层抽象

> 这是整个系统**最值钱**的设计。务必在重写中保留。

```
Matter（案件 / 项目）
  └── MatterProcedure（程序：一审 / 二审 / 仲裁 / 执行 …）
         └── MatterStage（工作阶段：立案 / 举证 / 开庭 / 判决 …）
                └── Task / Hearing / Deadline / Document
```

**为什么这样分**：律师视角下"青石建设诉华东置业"是**同一个案件**，无论经历一审、二审、再审——客户不变、合同通常一份贯穿。但每个程序有**独立的案号、法院、立案日、开庭与期限**，必须能分开管理。绝大多数老案管系统用"一案一案号"的扁平模型，这是它们的根本缺陷。

### 3.1 实体目录（概念层）

| 实体 | 含义 | 关键属性 / 不变量 |
|---|---|---|
| **Matter** 案件 | 一个法律争议或非诉项目的整体容器 | 编号、标题、类别、状态、案由、标的额、我方诉讼地位、收案日；挂客户/团队/当事人/财务/分成/沟通/时间线 |
| **MatterProcedure** 程序 | 同一争议下的一个程序 | 类型、参与方式(ENGAGED/INFORMATIONAL)、独立案号、办理机关、立案日、结案日、结果；**律师可自由选首个程序类型，不强制从一审开始** |
| **MatterStage** 工作阶段 | 程序内部的工作步骤 | 名称、顺序、起止；按程序类型套用默认模板 |
| **Party** 当事人 | 案件/收案的当事人（可无限增减） | 姓名、证件号、角色(我方/对方/第三人/共同诉讼/代理人/证人) |
| **ProcedureParty** 程序内地位 | 当事人在**某个程序**中的诉讼地位 | 同一人一审是"被告"、二审是"上诉人"、执行是"被执行人"——地位**下沉到程序** |
| **Client** 客户 | 委托方主体（自然人/公司/组织） | **不做客户合并**，以案件为中心；同一客户多次咨询分别建 Intake |
| **Contact** 联系人 | 机构客户下的联系人 | 姓名、职位、电话 |
| **Intake** 收案 | 咨询登记，正式案件的前身 | 状态机见 §5.1；转化后生成 Matter |
| **CauseOfAction** 案由 | 规范案由库 | 强制从官方库选，禁止随手填字符串（见 §7.1） |
| **ConflictCheck / ConflictHit** 冲突检索 | 利益冲突检索记录与命中 | 分级 + 结论（见 §6.2） |
| **Hearing** 开庭 | 挂在程序下 | 时间、法庭、法官、地址、联系人 |
| **Deadline** 期限 | 挂在程序下 | 类别、到期日、计算依据、提醒天数（见 §6.4） |
| **Note** 沟通记录 | 挂在案件下 | 渠道(电话/微信/邮件/会议/法院)、对象、内容、标签、附件 |
| **Document** 材料 | 案件材料 | 分类、版本、可加密、可归卷宗；可挂 Matter/Intake/Procedure |
| **DocumentFolder** 卷宗 | 案件内物理归档目录 | 按案件类别自动生成默认结构（见 §7.2） |
| **DocumentTemplate** 文档模板 | 系统/律所级文书模板 | 变量自动填充生成 docx（见 §5.5） |
| **Billing / FeeEntry / CommissionPlan** 财务 | 结算单 / 收付流水 / 分成方案 | 分成自动化见 §6.3 |
| **InvoiceRequest** 开票申请 | 开票工作流 | 状态机见 §5.4 |
| **SealRequest** 用章申请 | 用印审批工作流 | 状态机见 §5.3 |
| **Preservation** 财产保全 | 保全跟踪 | 期限/续保/到期预警（见 §6.5） |
| **SmsMessage** 法院短信 | 短信解析收件箱 | 解析→匹配案件→生成开庭/期限（见 §5.6） |
| **ArchiveRecord** 归档记录 | 结案归档 | 完整性校验（见 §6.6） |
| **AuditLog** 审计日志 | 操作留痕 | 只追加，不可业务删除 |
| **TimelineEvent** 时间线 | 全案事件聚合 | 系统自动 + 手动 |

### 3.2 关键结构性不变量

1. **一案多程序**：一个 Matter 含 ≥1 个 Procedure，可时序串接。
2. **程序参与方式决定聚合**：
   - `ENGAGED`（我方代理）→ 完整工作流，进日程/期限/任务等聚合视图。
   - `INFORMATIONAL`（前序参考，他人代理）→ 只存元数据（案号/法院/判决日/结果）用于答辩参考，**不进任何聚合视图**。
3. **诉讼地位下沉到程序**（ProcedureParty）——案件级 `Party.standing` 仅作首程序/兼容用途。
4. **当事人无限增减**：多原告、多被告、多第三人都用 Party 表达。
5. **多客户**：一案可多个原告共同委托。
6. **软删除**：Matter / Client / Document / Note 用 deletedAt，不物理删除。
7. **金额必须用定点小数**（Decimal），绝不用浮点。

## 4. 受控词表（领域知识）

> 这些枚举**本身就是律师实务知识**——"有哪些程序""每个程序有哪些合法诉讼地位""办理机关是谁"。是重写时必须 1:1 保留的领域资产。

### 4.1 案件类别 `MatterCategory`
`CIVIL_COMMERCIAL` 民商事 · `CRIMINAL` 刑事 · `ADMINISTRATIVE` 行政 · `NON_LITIGATION` 非诉 · `LEGAL_COUNSEL` 常年顾问 · `SPECIAL_PROJECT` 专项

### 4.2 程序类型 `ProcedureType` × 案件类别（联动映射）

| 类别 | 可选程序类型 |
|---|---|
| 民商事 | 一审 / 二审 / 再审审查 / 再审 / 发回重审(一审) / 发回重审(二审) / 民商事仲裁 / 劳动仲裁 / 申请撤销仲裁裁决 / 不予执行仲裁审查 / 强制执行 / 执行异议 / 检察监督 / 自定义 |
| 刑事 | 侦查 / 审查起诉 / 一审 / 二审 / 死刑复核 / 再审审查 / 再审 / 刑罚执行 / 减刑假释审查 / 检察监督 / 自定义 |
| 行政 | 行政复议 / 一审 / 二审 / 再审审查 / 再审 / 非诉行政执行 / 检察监督 / 自定义 |
| 非诉/顾问/专项 | 非诉项目阶段（自由命名）/ 自定义 |

### 4.3 办理机关（按程序类型映射）
- 侦查 → 公安局 / 监察委 / 国安局
- 审查起诉 / 检察监督 → 检察院
- 审判类（一审/二审/再审…）→ 法院
- 民商事仲裁 → 仲裁委员会；劳动仲裁 → 劳动人事争议仲裁委
- 行政复议 → 复议机关
- 刑罚执行 → 监狱 / 看守所

### 4.4 诉讼地位 `LitigationStanding` × 程序类型（联动）

| 程序类型 | 必备地位 | 可选地位 |
|---|---|---|
| 一审 / 重审一审 | 原告、被告 | 第三人、反诉原告、反诉被告 |
| 二审 / 重审二审 | 上诉人、被上诉人 | 第三人 |
| 再审审查 / 再审 / 检察监督 | 再审申请人、再审被申请人 | 第三人 |
| 执行 / 执行异议 | 申请执行人、被执行人 | 第三人 |
| 仲裁 / 劳动仲裁 | 仲裁申请人、仲裁被申请人 | 第三人 |
| 行政复议 | 复议申请人、复议被申请人 | 第三人 |
| 刑事各程序 | 被告人、被害人、自诉人、刑事附带民事原告 | — |
| 非诉 / 自定义 | 项目当事人 | — |

### 4.5 其它关键枚举
- **程序状态**：待启动 / 进行中 / 已结
- **程序结果**：胜诉 / 部分胜诉 / 败诉 / 调解 / 撤诉 / 驳回 / 完成(非诉) / 移送下一程序 / 其他
- **案件状态**：待启动首程序 / 办理中 / 暂停 / 已结案 / 已归档
- **期限类别**：诉讼时效 / 举证期限 / 上诉期限 / 履行期限 / 答辩期 / 申请执行期限 / 申请撤销仲裁期限 / 自定义
- **材料分类**：证据 / 诉讼文书 / 程序性材料 / 裁判文书 / 合同 / 其他
- **沟通渠道**：电话 / 微信 / 邮件 / 会议 / 法院 / 其他
- **收费模式** `FeeType`：一次性 / 分期 / 风险代理(纯后付) / 风险代理(部分后付) / 按小时
- **用章种类**：公章 / 合同专用章 / 财务专用章 / 法定代表人章 / 合同审核章
- **保全类型**：诉前 / 诉中 / 执行；**保全财产类型**：银行存款 / 房产 / 车辆 / 股权 / 知识产权 / 其他

## 5. 业务流程（状态机）

### 5.1 收案 → 正式案件（核心主流程）

**Intake 状态**：`INTAKE`(已咨询) → `PENDING_CONFIRMATION`(待审批) → `CONVERTED`(已转案) / `DECLINED`(不接案)

```
[任何角色] 提交收案（字段联动：类别→程序→诉讼地位；委托方可自由输入自动建档；可传合同）
   → status = PENDING_CONFIRMATION
[ADMIN / PRINCIPAL_LAWYER]
   ├─ 转为正式案件 → 生成 Matter（编号、套阶段模板、迁移合同 Document）+ Intake=CONVERTED
   └─ 不接案 → Intake=DECLINED + 记 declinedReason
```
**规则**：
- 审批权限**在服务端校验**，不只是 UI 隐藏。
- 合同迁移：Intake 关联的 Document 在转化时把归属改为新 Matter，但保留 intakeId 双向溯源。
- 标题可空，按 `{委托方} 与 {对方} {案由}纠纷` 自动生成。

### 5.2 多程序办理
- 新建案件时律师选**首个程序类型**（默认一审，可选任意），默认 ENGAGED。
- 详情页"+ 添加程序"：选类型 + 参与方式（我方代理 / 前序参考）。
- 仅 ENGAGED 程序的开庭/期限/任务进入日程与聚合视图。

### 5.3 用印审批 `SealRequest`
**状态**：`PENDING → APPROVED → STAMPED`；旁支 `REJECTED` / `CANCELLED`
```
[律师] 新建用章申请（章种类 + 可选关联案件 + 事由 + 待盖章稿[必传]）→ PENDING
[按章种类映射的审批人] ├─ 驳回→REJECTED（可引用旧 ID 重申请）└─ 通过→APPROVED
[行政/审批人] 上传盖章后扫描件[必补] → STAMPED
[申请人] 审批前可撤销 → CANCELLED
```
**规则**：审批人**按章种类映射角色**（公章/合同章/审核章→主任；财务章→财务；法定代表人章→Settings 指定的法定代表人本人；ADMIN 跨章可审）。盖章后扫描件**强制必传**才能进 STAMPED（合规留痕，事后无法补的环节卡死）。

### 5.4 开票工作流 `InvoiceRequest`
**状态**：`PENDING → APPROVED → ISSUED`；旁支 `REJECTED`
```
[案件 LEAD/CO_LEAD/ADMIN] 申请开票（金额+抬头+开票依据附件[必传]）→ PENDING
[FINANCE/ADMIN/PRINCIPAL_LAWYER] ├─ 驳回→REJECTED └─ 处理→上传电子发票→ISSUED
```

### 5.5 文书模板生成
选模板 → 自动拼装上下文（律所抬头/案件/客户/当事人/当前程序/主办律师/今日）→ 缺失变量行内补全（即写即回存源表）→ 渲染 docx → 下载 **并** 入库归到对应卷宗（存上下文快照便于复核/重生成）。

### 5.6 法院短信解析 `SmsMessage`
```
粘贴短信原文 → 本地正则解析(案号/法院/开庭时间/法官/上诉期/金额/URL)
   → 按案号反查 MatterProcedure 自动匹配案件
   → 一键生成 Hearing / Deadline（落到该案）→ processed=true
```
**规则**：解析纯本地正则（不依赖外部 AI，AI 仅作兜底增强）；案号正则需匹配 `(2026)沪01民终3520号` 一类格式。

### 5.7 结案归档
```
案件全部程序已结 → 填结案小结 → 完整性清单校验(§6.6) → 归档
   → 生成 ArchiveRecord + 案件转 ARCHIVED(只读) → 可导出单案 ZIP（材料 + 结构化 JSON）
```

## 6. 核心业务规则

### 6.1 案件编号生成
格式：`LL-{YYYY}-{CODE}-{4位流水}`，CODE = `CC`民商事 / `CR`刑事 / `AD`行政 / `NL`非诉 / `GC`顾问 / `SP`专项。
- 流水按 **年份+类别** 组合独立计数（如 `LL-2026-CC-0015`）。
- **计数器递增必须并发安全**（原子事务），避免并发收案重号。原实现用 Serializable 事务保证；新实现可用 DB 唯一约束 + 重试，或单点串行写。

### 6.2 利益冲突分级矩阵 ⭐（执业伦理红线）

按"**本次当事人角色** × **历史命中角色**"定级：

| 本次 | 历史 | 等级 | 含义 |
|---|---|---|---|
| 对方 | 我方客户 | 🔴 `BLOCKING` | 我们曾代理的人，现在成了对方——**绝对冲突** |
| 我方客户 | 对方 | 🟠 `HIGH` | 曾经的对手，现在要代理他 |
| 对方 | 对方 | 🟡 `LOW` | 历史诉讼对手再次出现 |
| 我方客户 | 我方客户 | 🟡 `LOW` | 回头客 |
| 第三人（任一侧） | — | 🟡 `MEDIUM` | 涉第三人 |

**加成规则**：命中**证件号精确匹配** → 等级 **+1 档**（封顶 BLOCKING）。

**检索范围**：全所历史（所有非删除案件的当事人 + 客户 + 客户↔案件反查），**不限于当前用户**。
⚠️ 原实现匹配较弱（仅 `contains` 子串、要求 ≥3 字、同名客户检测被禁用），重写应增强（见第 9 节）。

### 6.3 律师费分成自动化
- 案件配 `CommissionPlan`（每位受益人一个百分比，0–100；之和不强制=100，律所留存可隐含）。
- **每笔实收(RECEIVED)金额 X 入账时**：对每条 active plan 自动生成一条分成条目 `COMMISSION = -X × percent / 100`，挂在受益人下、链回父条目。
- **撤销实收**：级联删除其关联的分成子条目（事务内）。
- 百分比 0–100 校验，防止越界。
- ⚠️ 风险代理/计时收费目前只当一次性金额记账，无"结果触发后付/比例"或"工时累计"——重写应补（见第 9 节）。

### 6.4 期限与提醒
- 每条 Deadline 有 `category` / `dueAt` / `basis`(依据) / `remindDays`(提前几天提醒)。
- 提醒由定时任务每日扫描，对临近/逾期的期限生成通知。
- ⚠️ 原实现 `dueAt` **全靠手填**，`basis` 仅自由文本，**不自动推算法定期限**——这是最该补的能力（见第 9 节，附法定期限表）。

### 6.5 财产保全规则
- `expiryDate = startDate + duration(天)`，可手动覆盖。
- **默认期限按财产类型**（法律依据：最高法保全规定，需复核现行法）：

| 财产类型 | 默认天数 | 依据 |
|---|---|---|
| 银行存款 | 365（1 年） | 冻结银行存款等不超过 1 年 |
| 车辆 / 其他动产 | 730（2 年） | 查封扣押动产不超过 2 年 |
| 房产 / 股权 / 知识产权 | 1095（3 年） | 查封不动产及其他财产权不超过 3 年 |

- **续保**：`newExpiry > oldExpiry` 才允许；更新到期日，状态置 RENEWED；已解除(LIFTED)不可续。
- **到期预警**：提醒窗口默认 `[30, 15, 7, 3, 1]` 天。
- ⚠️ 原实现**到期预警未接线**（窗口字段存了但无定时任务扫描）——重写必须接上，否则"漏续保=冻结失效=执业事故"（见第 9 节）。

### 6.6 归档完整性清单（按类别）

归档时校验必备材料是否齐全：

| 类别 | 必备材料 |
|---|---|
| 诉讼（民/刑/行） | 委托代理合同、授权委托书、起诉状/答辩状/上诉状、证据目录材料、裁判文书、结案登记表、办案小结、卷宗封皮、卷宗目录 |
| 非诉 / 专项 | 委托代理合同、法律意见书、出具法律文件、结案登记表、办案小结、卷宗封皮、卷宗目录 |
| 常年顾问 | 法律顾问合同、出具法律意见汇总、期满总结/续约/终止说明、卷宗封皮、卷宗目录 |

**归档后只读**：ARCHIVED 案件的所有写操作在**服务端拦截**（不只是 UI 隐藏）。
⚠️ 原实现完整性校验**可被"强制归档"绕过**（缺料只警告不拦截）。重写建议：允许 override，但**要求填理由 + 主任审批**，而非裸放行（见第 9 节）。

## 7. 领域参考数据

### 7.1 规范案由库 `CauseOfAction` ⭐
**为什么要规范化**："建设工程施工合同纠纷"和"建工合同纠纷"必须是同一案由，否则统计/筛选/冲突全对不上。**强制从官方库选，禁止随手填字符串**（兜底自由文本仅极特殊场景，且标黄不计入统计）。

| 类别 | 来源 | 量级 |
|---|---|---|
| 民商事 | 最高法《民事案件案由规定》(2020 修正) | ~480 三级 + 少量四级（实测 seed 1055 条） |
| 刑事 | 《刑法》分则 + 历次修正案罪名 + 最高法确定罪名补充规定 | ~469 罪名（实测 511 条） |
| 行政 | 最高法《行政案件案由暂行规定》(2021) | ~80-100（实测 163 条） |

结构：树形（一级/二级/三级/四级），带官方编号、简称、拼音、关键词，支持模糊搜索与树形浏览。旧案由弃用时标 `active=false` 保留历史引用。

### 7.2 默认卷宗结构（新建案件自动生成）
- 民商事 / 行政：收案 / 立案 / 委托手续 / 证据 / 程序文书 / 庭审 / 裁判 / 结案
- 刑事：收案 / 委托手续 / 阅卷 / 会见 / 取证 / 庭前 / 庭审 / 判决与上诉 / 结案
- 非诉 / 顾问 / 专项：立项 / 调研 / 工作底稿 / 出具文件 / 归档

### 7.3 阶段模板
按程序类型预置一套默认工作阶段（如诉讼一审：立案 → 举证 → 开庭 → 判决），管理员可改。

## 9. 重写时应补的规则（原实现缺陷）

> 把之前专业体检发现的缺口转成「重写时的需求」。这几项恰是把系统从"记录工具"升级为"主动防错风控工具"的关键。

### 🔴 9.1 法定期限自动推算引擎（最高价值）
原系统只"存"期限不"算"期限。重写应按"**程序类型 + 触发事件 + 起算日**"自动推算法定期限并生成 Deadline。**以下为常见规则，须由执业律师按现行法复核：**

| 期限 | 周期 | 起算 |
|---|---|---|
| 上诉期（民事/行政·判决） | 15 日 | 裁判送达次日 |
| 上诉期（民事/行政·裁定） | 10 日 | 裁定送达次日 |
| 上诉期（刑事·判决/裁定） | 10 日 | 送达次日 |
| 答辩期 | 15 日（涉外 30 日） | 收到起诉状副本次日 |
| 举证期限 | 法院指定 | 按程序/法院通知 |
| 申请再审 | 6 个月 | 裁判生效后 / 特定情形知道之日起 |
| 申请强制执行 | 2 年 | 履行期限最后一日（分期则每期末日） |
| 普通诉讼时效 | 3 年 | 知道或应当知道权利受损及义务人之日 |
| 申请撤销仲裁裁决 | 6 个月 | 收到裁决书之日 |

实现要点：从开庭日/裁判日/送达日等事件触发，按上表推算，自动建 Deadline 并接入提醒；`basis` 记录推算依据（便于复核）。

### 🔴 9.2 接上保全到期预警（防执业事故）
为 Preservation 的提醒窗口 `[30,15,7,3,1]` 接定时扫描，临近到期生成通知并在工作台高亮。这是模块存在的全部意义，必须接线。

### 🟠 9.3 增强冲突检索召回
- 加真正的编辑距离 / 分词 / 别名（字号、简称、曾用名）匹配，覆盖"中建八局 vs 中国建筑第八工程局"这类。
- 恢复同名客户检测。
- 关联实体（股东/实控人/担保人）做交叉匹配。
- 保留原有的"角色×角色"分级矩阵（§6.2，这部分逻辑是对的）。

### 🟠 9.4 实现证件号脱敏
按角色在**服务端**返回掩码（含归档导出），仅 ADMIN/主办/协办见明文。涉《个人信息保护法》敏感个人信息合规。

### 🟡 9.5 风险代理 / 计时收费核算
- 风险代理：建模"结果触发后付 + 约定比例/封顶"，结案达成时按比例生成应收。
- 计时：工时累计 × 费率 → 应收。
- 区分 `COST`(诉讼费/差旅垫付) 与收入，实收统计不混入成本。

### 🟡 9.6 归档 override 加约束
完整性校验缺料时，允许强制归档但**要求填理由 + 主任审批**并记审计，替代裸放行。
